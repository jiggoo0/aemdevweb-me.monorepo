# Hydration: สถาปัตยกรรมเชื่อมต่อ Server และ Client ที่ผู้เชี่ยวชาญต้องวางระบบให้เนี้ยบที่สุด

ในโลกของ **React 19** และ **Next.js 16**, "Hydration" คือกระบวนการที่สถาปัตยกรรมของ React นำเอา HTML ที่เรนเดอร์มาจากเซิร์ฟเวอร์มา "ชุบชีวิต" ให้กลายเป็นแอปพลิเคชันที่โต้ตอบได้ที่ฝั่งเบราว์เซอร์ หากวางระบบผิดพลาดเพียงนิดเดียว จะเกิด **Hydration Error** ซึ่งส่งผลเสียต่อประสิทธิภาพและความน่าเชื่อถือทันที

## 1. สาเหตุหลักที่ทำให้การวางระบบผิดพลาด (Hydration Mismatch)

ส่วนใหญ่เกิดจากการใช้ข้อมูลที่ "ไม่แน่นอน" ในขณะเรนเดอร์โดยขาดการวางระบบที่รัดกุม เช่น:

- **Dates & Times:** การใช้ `new Date()` ที่เซิร์ฟเวอร์และเบราว์เซอร์จะได้เวลาต่างกันเสมอ หากไม่วางระบบป้องกัน
- **Browser-only APIs:** การเรียกใช้ `window` หรือ `localStorage` โดยไม่มีการเช็คสถานะการเมาท์ (Mount) จากผู้เชี่ยวชาญ
- **Random Values:** การใช้ `Math.random()` ที่สุ่มค่าออกมาไม่ตรงกันทั้งสองฝั่งโดยไม่มีมาตรฐานสากลรองรับ

## 2. กลยุทธ์การวางระบบระดับสถาปัตยกรรมเฉพาะทาง

ที่ **AEMDEVWEB**, ผู้เชี่ยวชาญของเราใช้มาตรฐานการเขียนโค้ดที่รัดกุมเพื่อยกระดับความเสถียร:

### การวางระบบ `suppressHydrationWarning` อย่างถูกวิธี

สำหรับข้อมูลที่จำเป็นต้องต่างกัน (เช่น เวลาปัจจุบัน) เราจะวางระบบโดยใช้ Attribute นี้เฉพาะจุดเพื่อบอก React ว่า "เรารู้ตัวนะ" เพื่อรักษาประสิทธิภาพและไม่ให้เกิด Error บล็อกการทำงานส่วนอื่น

### Two-pass Rendering: มาตรฐานสากลเพื่อความปลอดภัย

เราวางระบบโดยใช้ `useEffect` เพื่ออัปเดตข้อมูลฝั่ง Client หลังจากที่คอมโพเนนต์เมาท์เสร็จแล้วเท่านั้น เพื่อรับประกันว่าสถาปัตยกรรม HTML ชุดแรกที่ผู้ใช้เห็นจะตรงกับ Server 100%

```tsx
const [isMounted, setIsMounted] = useState(false);
useEffect(() => setIsMounted(true), []);

if (!isMounted) return <Skeleton />; // แสดงผลนิ่งๆ ตามมาตรฐานสากลก่อนชุบชีวิต
```

## 3. ผลกระทบต่อ SEO และประสิทธิภาพ Core Web Vitals

หากหน้าเว็บเกิด Hydration Error บ่อยๆ เบราว์เซอร์จะต้องเสียเวลาเรนเดอร์ใหม่ทั้งหมด (Bailout) ซึ่งจะทำให้ค่า **LCP (Largest Contentful Paint)** ช้าลง และอาจทำให้ Google Bot ประเมินว่าสถาปัตยกรรมเว็บไซต์ไม่มีความเสถียร
